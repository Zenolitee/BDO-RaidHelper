# Security

## Trust Boundaries

NW Helper handles Discord bot credentials, OAuth credentials, server membership data, dashboard sessions, and roster state. Run it as a server-side application. Never expose `.env`, Supabase service-role keys, or Discord bot tokens to browser code.

## Discord Permissions

- Slash commands are registered with Administrator default permissions.
- Runtime handlers also require Discord Administrator permission, so command visibility is not the only enforcement layer.
- `/set-nwchannel` must run inside a guild text channel.
- Signup buttons verify that the stored event guild matches the guild where the button was used.
- Legacy posted-roster management handlers for `Post now`, `Refresh`, and `Close` require Administrator permission or `OFFICER_ROLE_ID`. Current renderers expose `Post now` only from the Administrator-only `/event show` preview.
- The web dashboard only retains Discord servers where the logged-in user is an Administrator and the bot is installed.

The bot invite generated by the home page requests the permissions needed to view channels, send messages, embed links, read message history, use external emojis, mention configured roles, and use slash commands.

## Dashboard Authentication

- OAuth requests only `identify guilds`.
- OAuth state values are random, expire after ten minutes, and are consumed on callback.
- Dashboard session IDs are opaque 32-byte random values.
- Cookies are `HttpOnly`, `SameSite=Lax`, scoped to `/`, and marked `Secure` when `PUBLIC_BASE_URL` starts with `https://`.
- Session lifetime is 24 hours.
- State-changing dashboard forms require a per-session CSRF token.
- Supabase session storage hashes the raw cookie token with SHA-256 before persistence.
- Discord OAuth access tokens are used only to fetch the login profile and guild list; they are not stored in the session.

## HTTP Headers

The dashboard removes `X-Powered-By`, disables caching, and sets:

- `Content-Security-Policy`
- `Referrer-Policy: no-referrer`
- `X-Content-Type-Options: nosniff`
- `X-Frame-Options: DENY`

Rendered user and Discord values are escaped before HTML insertion.

## Storage

JSON mode writes a temporary file and renames it into place. `EventStore` serializes operations with an in-process promise queue. This is suitable for one application process.

For production and restart persistence, configure Supabase and use `SUPABASE_SERVICE_ROLE_KEY`. Both documented tables enable row level security, so a service-role key is expected for server-side access.

## Secret Handling

- Keep `.env` private.
- Keep `railway-variables*.txt` private and untracked.
- Never commit or paste `DISCORD_TOKEN`, `DISCORD_CLIENT_SECRET`, or `SUPABASE_SERVICE_ROLE_KEY`.
- Rotate a Discord bot token immediately if it has been exposed.
- Use deployment-provider secret variables instead of checked-in environment files.

The repository `.gitignore` already excludes `.env`, `railway-variables*.txt`, logs, generated output, and local data.

## Operational Limits

- Wizard state and OAuth state are process-local and disappear on restart.
- Dashboard sessions are also process-local unless Supabase service-role session storage is enabled.
- JSON storage is not designed for multiple application replicas.
- The public event detail page is accessible to anyone who knows an event ID. It contains roster display data by design.
